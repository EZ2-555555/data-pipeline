\section{Key Enhancements Beyond the Original Proposal}
\label{sec:enhancements}

Two significant enhancements were made beyond the original proposal: (1) data source coverage was expanded from two to five sources (ArXiv, Hacker News, DEV.to, GitHub Trending, RSS feeds) to improve corpus diversity and temporal coverage; and (2) the preprocessing pipeline was extended from basic HTML stripping to a 17-step normalization pipeline (validated by 18 dedicated unit tests) to handle the diverse content formats encountered across sources, particularly Markdown-heavy content from DEV.to and GitHub. All five ingesters share the same idempotent, SHA-256-based deduplication architecture.

Additionally, nine research-backed retrieval improvements were applied post-proposal, including: weighted Reciprocal Rank Fusion (vector~0.50, BM25~0.35, recency~0.15), improved BM25 tokenization with stop-word removal, BM25-only candidate quality gate, asymmetric candidate limits, post-fusion score gate, Shewhart $3\sigma$ minimum standard deviation floor for drift detection, RAGAS evaluation threading fix, cross-encoder reranking (ms-marco-MiniLM-L-6-v2), and NaN retry pass for RAGAS scoring. These improvements are documented in the project's engineering change log and collectively produced a composite quality swing of $+0.069$---from trailing the baseline by $-0.022$ to leading by $+0.047$.

\section{Preprocessing Pipeline Architecture}
\label{sec:preprocessing_detail}

The 17-step text normalization pipeline is applied to all ingested documents before chunking to handle the heterogeneous content formats across five data sources. The steps are sequenced to prevent cascading errors (e.g., stripping HTML tags before removing entity references):

\begin{enumerate}[leftmargin=0.5in, font=\small]
  \item HTML entity unescaping (\texttt{\&nbsp;} $\to$ space)
  \item HTML comment removal (\texttt{<!-- -->})
  \item HTML tag stripping (e.g., \texttt{<p>}, \texttt{<div>}, \texttt{<span>})
  \item Fenced code block removal (\texttt{```...```})
  \item Inline code removal (\texttt{`...`})
  \item Markdown image/link reference removal (\texttt{![alt](url)}, \texttt{[text](url)})
  \item URL removal (http, https, ftp schemes)
  \item Markdown heading removal (\texttt{\# \#\# \#\#\#})
  \item Markdown bold/italic marker removal (\texttt{**}, \texttt{*}, \texttt{\_\_}, \texttt{\_})
  \item Markdown strikethrough removal (\texttt{~~text~~})
  \item Markdown horizontal rule removal (\texttt{---}, \texttt{***})
  \item Markdown blockquote marker removal (\texttt{>})
  \item Emoji and pictographic symbol removal (Unicode ranges for emoticons, transport, supplemental symbols, regional indicators, dingbats, variation selectors)
  \item Unicode control character removal (bytes \texttt{0x00--0x1F}, zero-width joiners, byte-order marks)
  \item Repeated punctuation collapse (e.g., \texttt{!!!} $\to$ \texttt{!})
  \item Unicode whitespace normalization (\texttt{0xA0} $\to$ space)
  \item General whitespace collapse (multiple consecutive spaces, tabs, newlines)
\end{enumerate}

Following normalization, text is split into overlapping token-based chunks using the GPT-4 tokenizer (\texttt{cl100k\_base} encoding from TikToken) with configurable chunk size and overlap. Default parameters are chunk size 500 tokens ($\approx$400 words) and 50-token overlap, balancing context coherence against retrieval candidate diversity. The chunking operation produces 6{,}220 indexed chunks from the 3{,}984 ingested documents.

\subsection*{Key Deployment Characteristics}

The key deployment characteristics are:

\begin{itemize}[leftmargin=0.5in]
  \item \textbf{RDS PostgreSQL instead of Aurora Serverless.} Standard RDS PostgreSQL \texttt{db.t3.micro} (PostgreSQL 16.6, 20GB gp2) replaces the originally proposed Aurora Serverless to stay within budget. The pgvector extension and HNSW indexing remain fully functional.
  \item \textbf{Groq as primary LLM backend.} Groq free-tier API (Llama 3.1 8B Instant) serves as the primary backend, with Amazon Bedrock (Nova Micro), Ollama, and HuggingFace as fallbacks. The auto-fallback chain ensures generation availability.
  \item \textbf{Default VPC with security groups.} The SAM template uses the default VPC with two subnets. RDS is configured as publicly accessible with security group restrictions; custom VPC migration is identified as a future improvement.
  \item \textbf{Fine-grained IAM roles.} Custom IAM role definitions provide least-privilege access for each Lambda function.
  \item \textbf{Cost governance.} Cost governance is implemented primarily through programmatic AWS Cost Explorer checks in the inference path, supplemented by CloudWatch metrics, SNS-based alerting, and backend fallback logic. Native AWS Budgets was considered but is not part of the active Free Tier SAM template; budget enforcement relies on the runtime Cost Explorer guard and the automatic LLM backend fallback chain.
  \item \textbf{LLM backend set to \texttt{groq}.} The \texttt{LLM\_BACKEND} environment variable defaults to \texttt{groq}, with Amazon Bedrock and Ollama retained as fallbacks.
\end{itemize}

All infrastructure is defined in a SAM template (\texttt{template-freetier.yaml}) and configuration file (\texttt{samconfig-freetier.toml}), ensuring reproducible and consistent deployments from the same codebase. Step-by-step reproduction instructions are provided in Appendix~\ref{app:setup}.

\section{Experiment Tracking and Result Versioning}
\label{sec:experiment_tracking}

Experiment results are tracked through structured JSON artifacts versioned alongside the source code in Git. Each evaluation run produces three output files: \texttt{eval\_summary.json} (aggregate metrics, latency statistics, and cost projections for both configurations), \texttt{ragas\_scores.json} (per-query RAGAS faithfulness, answer relevancy, and context precision scores for all 100 observations), and \texttt{raw\_results.json} (complete per-query records including retrieved contexts, generated answers, citation grounding details, and timing breakdowns). This file-based approach was chosen over a dedicated experiment tracking platform (e.g., MLflow, Weights \& Biases) for three reasons: (1) the evaluation compares exactly two fixed configurations (baseline vs.\ hybrid) rather than a large hyperparameter sweep requiring run management; (2) JSON artifacts are directly consumable by the statistical analysis and visualization scripts without additional API dependencies; and (3) Git versioning provides full traceability of which code revision produced which results, with \texttt{git diff} enabling precise identification of changes between evaluation runs. The design-phase grid search results (\texttt{stale-results-hybrid/}) are similarly retained under version control to document the design evolution from linear scoring to RRF-based fusion.

\section{Engineering Observations}
\label{sec:eng_obs}

Several non-obvious findings arose during implementation that informed the final design:

\begin{itemize}[leftmargin=0.5in]
  \item \textbf{Lambda cold-start and MiniLM.} Initial cold-start delays of 8--12\,s were resolved by a lazy-initialized singleton and 1{,}024\,MB memory allocation, reducing model load to $<$2\,s on warm invocations.

  \item \textbf{SHA-256 deduplication.} The \texttt{ON CONFLICT DO NOTHING} strategy proved fully effective; ArXiv produced the most hash collisions ($\sim$15--20\% per cycle) due to preprint updates.

  \item \textbf{LocalStack fidelity.} S3 and SQS operations matched production behavior, enabling integration testing without AWS credentials. CloudWatch metrics required cloud deployment to verify.

  \item \textbf{HuggingFace rate limits.} The free-tier rate limit ($\sim$30 req/min) triggered the Ollama fallback during sustained evaluation runs, confirming the fallback chain operates correctly.

  \item \textbf{pgvector HNSW index build.} Index rebuilds took 45--90\,s on db.t3.micro for corpora above $\sim$10k chunks---acceptable for batch but would need management for real-time updates.
\end{itemize}

\section{Hybrid Retrieval: BM25 Index Caching \& Architectural Details}
\label{sec:bm25_caching}

The BM25 component of hybrid retrieval employs module-level index caching with automatic periodic refresh. The full-corpus BM25 index is constructed from all indexed chunks during the initial load and then cached in process memory. The cache is automatically invalidated and reconstructed every 30 minutes to incorporate newly ingested documents while minimizing per-query recomputation overhead. This caching strategy reduces mean BM25 retrieval latency by $\sim$200\,ms (per-query computation ~$O(N)$ tokens against the corpus) but introduces a bounded staleness: queries execute against corpus state from at most 30 minutes prior. For drift detection validation runs, probe queries execute against potentially cached BM25 results; sensitivity analysis confirms that cache age induces <1\% variation in mean cosine similarity measurements, well within the 10\% drift threshold margin.

The BM25 tokenizer (class \texttt{BM25Okapi}) is independent of the query processing tokenizer to ensure consistent vocabulary across corpus indexing and retrieval. Token filtering removes 125 domain-adapted stopwords (including common tech-domain terms like ``tech'', ``technology'', ``article'', ``data''), enforces minimum token length of 2 characters, and normalizes the text to lowercase before tokenization. This design choice prevents false positive matches on ubiquitous terms while preserving domain-specific signal.

\section{Ingestion Scheduler \& Orchestration}
\label{sec:scheduler}

TechPulse implements two ingestion orchestration modes: (1) AWS EventBridge-based scheduling for cloud deployments, and (2) local polling-based scheduler for Docker Compose environments.

\noindent\textbf{AWS EventBridge:} On AWS, a CloudWatch Events rule (cron expression \texttt{cron(0 */6 * * ? *)}) triggers the ingestion Lambda every 6 hours (00:00, 06:00, 12:00, 18:00 UTC). Each invocation sequentially calls all five ingesters (ArXiv, Hacker News, DEV.to, GitHub Trending, RSS) within a single Lambda function, minimizing cold-start overhead. The 6-hour cycle was determined empirically to balance corpus freshness (query latency impact $<$100\,ms for 6-hour-old documents in the tech domain) against AWS Lambda free-tier invocation limits (1 million/month, ~4 invocations/day).

\noindent\textbf{Local Scheduler:} In Docker Compose development, a dedicated \texttt{scheduler} service implements polling-based orchestration with source-specific refresh intervals: ArXiv every 12 hours (lower volume), and Hacker News, DEV.to, GitHub, RSS every 30 minutes (higher velocity signals). This asymmetric scheduling reflects domain knowledge about source update patterns.

\section{API Endpoints \& Request Handling}
\label{sec:api_endpoints}

The FastAPI application exposes four primary endpoints:

\begin{enumerate}[leftmargin=0.5in]
  \item \texttt{GET /health} --- Deep health check returning the status of database, S3, SQS, and configured LLM backend (response time: $<$2\,s).
  
  \item \texttt{POST /ask} --- Main query endpoint. Accepts \texttt{query} (string), \texttt{mode} (``baseline'' or ``hybrid''), and optional \texttt{top\_k} (default 5). Returns ranked passages, generated answer (if LLM available), per-stage latencies, and citation metadata. Enforced rate limit: 20 requests/minute per IP address (SlowAPI).
  
  \item \texttt{POST /drift} --- Manual drift detection trigger. Executes the nine simulated drift scenarios (corpus drift, query drift, embedding shift, LLM change, source imbalance, temporal shift, semantic shift, and label noise) and returns alerting verdicts.
  
  \item \texttt{GET /dashboard/insights} --- Complex aggregation endpoint returning dashboard data: trending topics (keywords + growth metrics), 30-day mention time-series, source distribution, and live highlight snippets (top 5 by composite score). Optional query parameters: \texttt{days} (7--60, default 30) and \texttt{sources} (comma-separated filter over ARXIV, HN, DEVTO, GITHUB, RSS).
\end{enumerate}

Rate limiting is implemented via SlowAPI (dependency: \texttt{slowapi}) to provide basic DOS protection and fair resource allocation in shared environments. The 20 req/min limit on \texttt{/ask} is conservative; monitoring during evaluation runs showed peak throughput of $\sim$2--3 queries/min from a single user.

\section{Citation Grounding Verification}
\label{sec:citation_grounding}

The three-layer hallucination verification strategy includes an output-layer citation grounding check that validates the generated response's reference completeness. The verification logic:

\begin{enumerate}[leftmargin=0.5in]
  \item \textbf{Sentence splitting:} The response is split on sentence-ending punctuation using regex \texttt{r'(?<=[.!?]){\textbackslash}s+'}, producing atomic sentential units.
  
  \item \textbf{Citation detection:} Each sentence is checked for presence of citation markers matching the pattern \texttt{r'{\textbackslash}[Source{\textbackslash}s+{\textbackslash}d+{\textbackslash}]'}, where the captured index $N$ must satisfy $1 \leq N \leq \text{num\_sources}$.
  
  \item \textbf{Citation ratio computation:} The fraction $R = \text{cited\_sentences} / \text{total\_sentences}$ is computed and compared against a configurable threshold \texttt{CITATION\_GROUNDING\_THRESHOLD} (default 0.0 for local dev, recommended $\geq 0.5$ for production with capable LLMs).
  
  \item \textbf{Hallucinated reference detection:} Invalid source indices (e.g., \texttt{[Source 12]} when only 5 sources are available) trigger a hallucination flag.
\end{enumerate}

Responses flagged as under-grounded (citation ratio $< $ threshold) or containing invalid references are recorded in CloudWatch as \texttt{HallucinationFlags} metrics for operational monitoring. During evaluation, the RAGAS-computed ``Citation Grounding'' metric (Section~5) correlates with but is independent of this output-layer check, providing complementary signal for model quality assessment.

In addition to the primary search interface, a comprehensive dashboard provides real-time system monitoring and evaluation results visualization. The dashboard aggregates the following functional components:

  \item \textbf{Pipeline Architecture Diagram.} A sequential six-stage flowchart visualizing the ingestion-to-generation pipeline (Data Sources → Ingestion → Embedding → pgvector DB → Hybrid Retrieval → LLM Answer Generation), providing non-technical stakeholders with system architecture intuition.

  \item \textbf{Evaluation Results Aggregation.} Displays all evaluation metrics computed in Chapter~5:
  \begin{itemize}
    \item Radar plot comparing baseline and hybrid RAGAS scores (five dimensions: Faithfulness, Answer Relevancy, Context Precision, Citation Grounding, Composite).

    \item Line chart of sensitivity analysis results, showing composite score variation across $\alpha \in [0.0, 1.0]$ (semantic weight), highlighting the optimal $\alpha = 0.7$ configuration.

  \end{itemize}

  \item \textbf{Multi-Source Attribution.} All highlights include explicit source attribution and clickable source chips at the dashboard base, allowing rapid filtering by data origin and validation that the system ingests from all five specified sources.
\end{enumerate}

The dashboard serves two purposes: (1) communicating system capabilities and results to non-technical stakeholders (e.g., project advisors, peer reviewers) through summary statistics and visualizations, and (2) enabling real-time operational monitoring of ingestion freshness and source distribution balance. Interactive features (source filtering, clickable external links, animated metrics) provide immediate feedback to users exploring the system's outputs.
