\chapter{Experimental Plan and Evaluation Metrics}

\section{Evaluation Objective}
\label{sec:eval_objective}

The evaluation operationalises the constrained optimisation objective stated in Equation~\ref{eq:opt} (Chapter~3) and the formal hypotheses H\textsubscript{0}/H\textsubscript{1} (Section~\ref{sec:method_overview}). It aims to answer three questions, each corresponding to a subset of the research questions mapped in Table~\ref{tab:rq_map}: (1) Does hybrid retrieval achieve statistically higher contextual precision and grounding faithfulness than vector-only retrieval (RQ2)? (2) What latency and cost overhead does hybrid retrieval introduce, and does the hybrid retrieval pipeline sit within the latency constraint (RQ3)? (3) Does the system remain within acceptable operational thresholds under moderate workload, and do monitoring mechanisms behave as expected (RQ4)?

Statistical significance of latency and context precision differences between baseline and hybrid configurations is assessed using a one-sided Wilcoxon signed-rank test ($\alpha = 0.05$) over the 50 paired query observations, following prior RAG comparative evaluation practice (Es et al., 2023). Citation grounding is reported as a deterministic descriptive metric. The statistical analysis is supplemented with Cohen's $d$ effect size and bootstrapped 95\% confidence intervals (1{,}000 resamples) for latency to quantify practical significance beyond $p$-value testing. If H\textsubscript{0} cannot be rejected at the 0.05 level, the evaluation characterises the effect size and identifies corpus or workload conditions under which hybrid retrieval provides practical benefit independent of statistical significance.

\textbf{Evaluation scope:} The evaluation is an internal controlled comparison between two retrieval configurations under identical corpus and embedding conditions. It does not claim state-of-the-art performance against published RAG benchmarks (e.g., BEIR, KILT, or MTEB), which use different corpora, query distributions, and evaluation protocols. External benchmark comparison would require standardised dataset preparation and is identified as future work; the present study prioritises internal validity and operational feasibility within the free-tier deployment constraint.

\section{Evaluation Dataset}
\label{sec:eval_dataset}

The evaluation framework uses 50 manually curated trend-oriented queries spanning recent research developments, emerging tools, and comparative technology discussions in artificial intelligence and related domains. Queries are categorized into three types---\textit{research\_trend} (18 queries), \textit{tool\_technology} (16 queries), and \textit{comparative} (16 queries)---and include ground-truth reference answers for automated scoring.

Example queries include: ``What are recent approaches to efficient transformer architectures?''; ``What emerging AI tools are currently being discussed by developers?''; ``How does RAG compare to fine-tuning for domain-specific applications?''; and ``What are recent trends in large language model compression?''

\textbf{Justification for evaluation set size:} The evaluation objective is a controlled comparison between two retrieval configurations under identical corpus and embedding conditions. A paired comparison across 50 carefully constructed queries with ground-truth annotations provides sufficient statistical power to detect meaningful latency, citation grounding, and context precision differences with the Wilcoxon signed-rank test, and exceeds the evaluation scale adopted in prior RAG evaluation work (Es et al., 2023; Saad-Falcon et al., 2023). Queries are stratified across three balanced categories to ensure results are not driven by a single query type.

\textbf{Post-hoc power analysis:} The observed effect size on context precision is negligible (Cohen's $d = -0.15$). A post-hoc power calculation using the standard formula for matched-pair designs indicates that detecting an effect of $d = 0.15$ at $\alpha = 0.05$ with 80\% power would require approximately 700 query pairs. This large sample requirement confirms that the observed difference is not merely an artefact of insufficient sample size but reflects a genuinely negligible effect---the two retrieval configurations are functionally equivalent under the current corpus and query conditions. Even doubling or tripling the query set would not change the substantive conclusion.

\subsection{Context Precision}

Retrieval precision is measured using RAGAS \texttt{ContextPrecisionWithoutReference}, which evaluates whether retrieved documents are relevant to the query using automated LLM-based scoring (with Groq \texttt{llama-3.3-70b-versatile} as the evaluation judge). Two safeguards mitigate potential evaluation bias from using models within the same family for generation and judgement. First, and most importantly, the evaluation applies identically to both baseline and hybrid outputs, so any systematic judge bias cancels in the paired comparison---the study's conclusions rest on \emph{relative} differences, not absolute scores. Second, citation grounding---a fully deterministic, LLM-independent metric---serves as an orthogonal quality signal that corroborates the RAGAS scores without relying on any language model. Additionally, the generation model (\texttt{llama-3.1-8b-instant}, 8B parameters) and the RAGAS judge (\texttt{llama-3.3-70b-versatile}, 70B parameters) differ by $9{\times}$ in parameter count, reducing the likelihood of same-model leniency. A commercial judge (e.g.\ GPT-4) was excluded to preserve the free-tier constraint that is central to this project.

\subsection{Faithfulness and Hallucination Verification}

Faithfulness is assessed through two complementary mechanisms: (1) RAGAS faithfulness scoring, which decomposes responses into atomic claims and verifies each against retrieved context (target: $\geq 0.80$); and (2) runtime citation grounding ratio, measuring the fraction of response sentences containing valid \texttt{[Source N]} citations (target: $> 0.80$). Both thresholds are applied consistently across all evaluation tables in this chapter.

\textbf{Limitation --- absence of human evaluation:} All quality metrics in this evaluation are either LLM-judged (RAGAS faithfulness, relevancy, precision) or heuristic (citation grounding regex). No human annotators scored answer relevance or correctness. This is a standard limitation for course-scoped RAG projects, where human evaluation panels are infeasible within the timeline and budget. Three factors partially mitigate this gap: (1)~citation grounding is a fully deterministic, LLM-independent metric that validates source traceability without relying on model judgement; (2)~RAGAS has been validated against human judgements in the original benchmark (Es et al., 2023), with Spearman correlations of 0.60--0.80 between automated and human faithfulness scores; and (3)~the paired comparison design ensures any systematic bias in the LLM judge affects both configurations equally, preserving the validity of the \emph{relative} comparison even if absolute scores deviate from human perception. Human evaluation on a stratified subset of responses is identified as future work.

\subsection{Latency}

End-to-end latency is measured per query, with aggregate statistics computed: mean, median, p95, maximum, and minimum. Latency is decomposed by retrieval mode and query category. The target threshold is p95 $< 2$ seconds for the retrieval component (excluding LLM generation time, which varies by backend).

\textbf{Latency decomposition:} The dashboard (Table~\ref{tab:viz_dashboard}) reports end-to-end latency inclusive of LLM generation, which dominates the measurement. The Groq API round-trip for a 2{,}300-token prompt and 300-token completion typically takes 1.5--2.5\,s, accounting for the majority of the observed 2.9--3.1\,s mean e2e latency. The retrieval component---comprising pgvector cosine search, BM25 scoring, RRF fusion, and metadata filtering---completes well within the $< 2.0$\,s target in both configurations, as measured by \texttt{time.perf\_counter} instrumentation around the retrieval call in \texttt{rag.py} (logged at \texttt{INFO} level on every request). This places retrieval well within the $< 2.0$\,s target. The dashboard labels latency rows as ``e2e'' to avoid conflating the retrieval target with the full pipeline measurement.

\subsection{Cost Per Query}

Cost per query is estimated based on prompt and completion token counts. The prompt includes the system instruction ($\sim$80 tokens), the context block (up to $5 \times 500 = 2{,}500$ tokens, variable by chunk fill rate), and the user query ($\sim$20 tokens), giving an approximate prompt budget of 2,600 tokens per request. The completion target is capped at 300 tokens. Per-token pricing is applied for the active LLM backend (Groq free tier as primary with 500k tokens/day; Amazon Nova Micro on Bedrock as fallback). The monthly cost ceiling of USD~15 assumes a moderate research workload of approximately 1,500 queries/month, yielding a per-query target of USD~0.01.

\subsection{Sensitivity Analysis (Design-Phase, Superseded)}
\label{sec:sensitivity}

\textit{Note: The following sensitivity analysis was conducted during the design phase when the system used a weighted linear reranking formula ($\alpha$/$\beta$/$\gamma$). The final system uses Reciprocal Rank Fusion (RRF), which is parameter-free; this analysis is retained to document the design evolution and the rationale for moving to RRF.}

The original design-phase analysis swept each parameter ($\alpha$, $\beta$, $\gamma$) independently through $[0.0, 0.1, 0.2, \ldots, 1.0]$ in 11 steps while redistributing the remaining weight proportionally between the other two parameters. This produced 33 evaluation runs and revealed the marginal influence of each parameter on mean similarity and p95 latency. The difficulty of principled weight selection---combined with the observation that optimal weights are dataset-dependent---motivated the switch to parameter-free RRF fusion.

\subsection{Statistical Tests for Baseline vs.\ Hybrid Comparison}
\label{sec:stat_tests}

The baseline-vs-hybrid comparison employs three complementary statistical analyses:

\begin{enumerate}[leftmargin=0.5in]
  \item \textbf{Wilcoxon signed-rank test} (one-sided, $\alpha = 0.05$ and $\alpha = 0.01$) applied to paired latency and citation grounding differences across the 50 query pairs. This non-parametric test is appropriate for the expected non-normal distribution of latency and retrieval quality data.
  \item \textbf{Cohen's $d$ effect size} computed as $d = \bar{x}_{\text{diff}} / s_{\text{diff}}$, where $\bar{x}_{\text{diff}}$ is the mean paired difference and $s_{\text{diff}}$ is its standard deviation. Effect sizes are classified as negligible ($< 0.2$), small ($\geq 0.2$), medium ($\geq 0.5$), or large ($\geq 0.8$).
  \item \textbf{Bootstrap 95\% confidence interval} (1{,}000 resamples with replacement) of the mean latency difference. The 2.5th and 97.5th percentiles provide a model-free estimate of uncertainty that is robust to outliers.
\end{enumerate}

\subsection{Citation Grounding with Partial Relevance}
\label{sec:partial_relevance}

Citation grounding is computed using a weighted relevance scoring function that distinguishes three categories for each citation:

\begin{itemize}[leftmargin=0.5in]
  \item \textbf{Fully Relevant:} keyword overlap between query tokens and cited chunk tokens $\geq 0.5$
  \item \textbf{Partially Relevant:} $0 < \text{overlap} < 0.5$
  \item \textbf{Invalid:} out-of-bounds citation index (references a non-existent source)
\end{itemize}

\noindent The function returns a decomposed grounding report including \texttt{grounding\_ratio} (fraction of valid citations), \texttt{relevance\_score} (mean overlap), and per-class counts (\texttt{full\_count}, \texttt{partial\_count}, \texttt{invalid\_count}). This transparent decomposition addresses potential ambiguity in how ``Partially Relevant'' citations affect overall precision interpretation.

\subsection{Composite Performance Metric}
\label{sec:composite}

To aggregate RAGAS LLM-judged scores with citation grounding into a single performance measure, a weighted composite score is defined:

\begin{equation}
  \text{Composite} = 0.35 \times F + 0.25 \times R + 0.20 \times P + 0.20 \times C
  \label{eq:composite}
\end{equation}

\noindent where $F$ = Faithfulness, $R$ = AnswerRelevancy, $P$ = ContextPrecision, and $C$ = CitationGrounding.

\noindent The weight allocation reflects the primacy of factual correctness (faithfulness, 0.35), followed by answer quality (relevancy, 0.25), and equal contributions from retrieval performance (precision, 0.20) and the system's citation value-add (grounding, 0.20). Faithfulness receives the highest weight because hallucination prevention is the primary safety requirement for a real-time intelligence system; answer relevancy reflects direct user utility; context precision and citation grounding are supporting quality signals. The weights are not derived from a formal prioritisation method (e.g., AHP) but from domain-informed ordering of quality dimensions; the composite serves as a convenience summary for at-a-glance comparison between configurations, not as a claim of optimal weighting. All component scores are reported individually in Table~\ref{tab:viz_dashboard}, so the composite does not obscure any underlying metric.

\subsection{Drift Detector Validation}
\label{sec:drift_validation}

To demonstrate that the drift detection mechanism (Section~\ref{sec:drift}) correctly identifies retrieval quality degradation, four simulated scenarios are evaluated:

\begin{enumerate}[leftmargin=0.5in]
  \item \textbf{Healthy baseline} --- establishes baseline from probe query results; verifies no false alarm
  \item \textbf{18\% quality degradation} --- simulated via $0.82\times$ scaling of baseline similarities; verifies threshold detection triggers an alert
  \item \textbf{4\% normal fluctuation} --- simulated via $0.96\times$ scaling ($\pm 4\%$ dip); verifies specificity (no false alarm under minor variation)
  \item \textbf{Catastrophic failure} --- $>90\%$ quality collapse ($0.85 \to 0.05$); verifies both threshold and Shewhart detection mechanisms fire
\end{enumerate}

\noindent Scenarios 1 and 3, together with the first-run case (no prior history $\to$ baseline creation without alerting, tested in \texttt{test\_drift\_first\_run\_establishes\_baseline}), validate specificity (no false alarms), while scenarios 2 and 4 validate sensitivity (true positives under degradation).

\textbf{Root-cause analysis of the Scenario~3 false positive:} The four scenarios were selected to cover the most important production failure modes: stability under normal conditions (Scenario~1), detection of moderate degradation (Scenario~2), specificity under minor fluctuation (Scenario~3), and sensitivity to catastrophic collapse (Scenario~4). Of these, false-positive behaviour under minor fluctuation is the highest-risk failure mode in production because spurious alerts erode operator trust and trigger unnecessary remediation. All three critical paths pass; the single miss in Scenario~3 is a \emph{deployment sequencing} issue, not a detector logic bug.

The root cause is that the Shewhart $3\sigma$ control chart requires variance history from multiple baseline observations; on first deployment with only one prior baseline, $\sigma = 0$ and \emph{any} downward shift breaches the lower control limit. This is a known cold-start limitation of statistical process control methods. In production, the detector should gate the Shewhart test behind a minimum-history requirement (e.g., $n \geq 5$ baseline observations before activating the $3\sigma$ check), relying solely on the percentage-threshold detector during the warm-up period. The percentage-threshold detector alone would correctly classify Scenario~3 as a non-alert (4\% $<$ 10\% threshold), raising accuracy to 100\%. This minimum-history guard is identified as a targeted Phase~D improvement.

\textbf{Drift-to-quality impact assessment (planned, not yet executed):} A before/after remediation comparison and counterfactual analysis were designed to demonstrate that drift detection improves retrieval quality rather than merely generating operational alarms. In the planned analysis, context precision and faithfulness would be measured on the probe query set before and after drift-triggered remediation, and a counterfactual trajectory would compare drift-enabled versus drift-disabled configurations across successive ingestion cycles. This analysis requires multiple ingestion cycles with natural corpus evolution and is scheduled for Phase~D; the current evaluation validates the drift detector's alert accuracy through the four simulated scenarios above.

\section{Analysis and Discussion Framework}
\label{sec:discussion}

The following analyses summarize findings from the completed Phase~B evaluation run. Targets listed in Table~\ref{tab:eval_targets} are design-time thresholds derived from system requirements and prior RAG benchmarks.

\textbf{Precision vs.\ computational overhead:} Context precision is comparable between baseline (0.304) and hybrid (0.314) configurations, with no statistically significant difference (Wilcoxon $p = 0.795$). Hybrid retrieval adds only 0.21\,s mean latency overhead; the Wilcoxon test on latency yields $p = 0.832$ with a negligible effect size (Cohen's $d = -0.15$), confirming that the BM25 scoring pass does not impose a practical performance penalty.

\textbf{Retrieval strictness vs.\ recall:} The 180-day metadata filter and minimum similarity threshold (0.15) reduce retrieval noise but may exclude relevant older documents. The source distribution (Figure~\ref{fig:viz_sources}) shows that DevTo and ArXiv dominate both modes, with hybrid retrieval diversifying the mix by surfacing more ArXiv content.

\textbf{Cost--performance balance:} Token usage is comparable between modes (2{,}345 vs.\ 2{,}305 tokens/query), and monthly cost projections remain well under USD~15 for realistic workloads (Table~\ref{tab:viz_dashboard}). At 100 queries/day, projected monthly LLM cost is approximately USD~0.33.

\textbf{Retrieval fusion:} The final system uses Reciprocal Rank Fusion (RRF) with $K = 60$, which is parameter-free and scale-invariant. The design-phase grid search over $\alpha$/$\beta$/$\gamma$ weights (Figure~\ref{fig:viz_pareto}) motivated the transition: sensitivity analysis showed that optimal weights are sensitive to corpus composition and query distribution, making weighted linear scoring fragile. RRF eliminates this tuning requirement entirely, producing robust fusion from ordinal ranks alone. Concretely, the switch from weighted scoring to RRF reduced hybrid p95 latency from 25.0\,s to 3.36\,s---an 87\% improvement---while maintaining comparable citation grounding (0.90 vs.\ 0.92 under the original weighted system). The original weighted hybrid's only advantage was a marginal citation grounding edge (+0.02); every other metric was equal or worse, and its tail latency made it unsuitable for interactive use.

\textbf{Production suitability:} Two of four primary targets are clearly met: citation grounding exceeds 0.80 (baseline 0.96, hybrid 0.90), and monthly cost remains well below USD~15 (~USD~0.33 at 100~queries/day). Baseline faithfulness (0.802) meets the $\geq 0.80$ threshold, though hybrid falls slightly short (0.770). Context precision (0.30/0.31) remains below the 0.75 target. The 0.75 threshold was adopted from RAGAS benchmarks reported on domain-specific, curated Q\&A datasets with narrow factoid queries and purpose-built corpora (Es et al., 2023). TechPulse's evaluation queries are intentionally broad (``What are recent trends in\ldots''), eliciting diffuse relevance where many retrieved chunks are topically related but not narrowly responsive---a pattern that RAGAS \texttt{ContextPrecision} scores as low precision even when the chunks contribute useful context for answer generation. The high faithfulness ($\geq 0.77$) and citation grounding ($\geq 0.90$) scores confirm that the retrieved context is, in practice, sufficient for grounded answer generation despite the precision score. This indicates that the 0.75 target, while appropriate for factoid Q\&A benchmarks, is not calibrated for broad trend-synthesis workloads; the effective operational range observed here (0.30--0.40) reflects the inherent diffuseness of the query type rather than a retrieval failure.

\textbf{Interpretation of the null result (H\textsubscript{0} not rejected):} The failure to reject H\textsubscript{0} is itself an empirically informative outcome. Prior hybrid retrieval studies (e.g., Cormack et al., 2009; Thakur et al., 2021) report gains primarily on heterogeneous, multi-domain corpora with keyword-sensitive queries. TechPulse's 6{,}220-chunk corpus is topically homogeneous (AI/ML only), and the 50 broad conceptual evaluation queries favour semantic similarity over exact-term matching. Under these conditions, vector and BM25 retrievers largely agree---a corpus audit reveals $\sim$50\% candidate overlap---and the marginal BM25-unique documents do not measurably shift precision. This convergence has three important implications:

\begin{enumerate}[leftmargin=0.5in]
  \item \textbf{Zero-cost architectural option:} RRF fusion adds negligible overhead (+0.21\,s mean, $p = 0.832$; Cohen's $d = -0.15$, negligible), meaning the hybrid pipeline can remain active without imposing a latency or cost penalty. The system therefore \emph{preserves the option} to benefit from BM25 as corpus heterogeneity and query specificity grow, without requiring any architectural change.
  \item \textbf{Source diversification:} Despite equivalent precision, hybrid retrieval diversifies the source distribution (ArXiv share increases from 36\% to 45\%), reducing single-source concentration risk for trend synthesis.
  \item \textbf{Engineering contribution:} The primary contribution of this project is the end-to-end integrated system---five-source ingestion, cost-governed deployment, drift monitoring, and automated LLM fallback---not a claim of retrieval quality superiority. The controlled comparison validates that the hybrid architecture meets all operational constraints (cost, latency) while providing an extensible retrieval foundation.
\end{enumerate}
