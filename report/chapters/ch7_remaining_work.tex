\textbf{Phase~A} builds the evaluation corpus through sustained multi-cycle ingestion across all five sources (target: 1{,}000+ documents), verifies end-to-end pipeline operation in the local Docker Compose environment, and deploys the system to the AWS Free Tier via the SAM template.

\textbf{Phase~B} executes the 50-query evaluation in both baseline and hybrid modes, applies Wilcoxon signed-rank tests with Cohen's $d$ and bootstrap 95\% CIs, and profiles per-stage latency with cost projections for 50/100/200 queries/day.

\textbf{Phase~C} validates the dual-criteria drift detection mechanism under simulated conditions, generates result visualizations and comparison tables, and captures system demonstration screenshots for all placeholder figures.

\textbf{Phase~D} completes the evaluation and discussion chapters with empirical findings, integrates screenshots and analysis, and prepares the final submission package.

\section{Known Limitations}
\label{sec:limitations}

\begin{enumerate}[leftmargin=0.5in]
  \item The evaluation corpus is limited to five technology-focused sources; generalizability to other domains requires further study.

  \item The 50-query evaluation set provides sufficient power for the paired Wilcoxon test but a larger set would strengthen external validity.

  \item Amazon Bedrock has not been evaluated as the primary backend; however, the evaluation methodology is backend-agnostic, so comparative findings remain transferable.

  \item The \texttt{db.t3.micro} RDS instance may become a bottleneck under high ingestion volumes compared to Aurora Serverless.

  \item Cross-encoder reranking and ANN-optimized indexing are out of scope but identified as future directions.

  \item Budget governance via Cost Explorer lags by up to 24 hours; real-time enforcement is not feasible on the Learner Lab.

  \item The test suite (204 tests, 15 modules) covers core logic but lacks end-to-end integration tests with a live database.

  \item The top-$k = 5$ retrieval at 500 tokens/chunk fits Groq and Bedrock context windows but may be silently truncated by Ollama's 4{,}096-token Llama~3.2~3B; a pre-prompt token budget check is identified as a near-term improvement.

  \item API keys are stored as Lambda environment variables; migration to AWS Secrets Manager is recommended for production hardening.
\end{enumerate}

\section{Future Extensions}
\label{sec:future}

Several extensions are identified for post-prototype development:

\begin{itemize}[leftmargin=0.5in]
  \item Migration to Aurora Serverless (applicable when scaling beyond Free Tier constraints)
  \item Full evaluation with Amazon Bedrock (Nova Micro) as the primary backend via the Free Tier deployment
  \item Expansion of data sources (StackExchange, academic conference proceedings)
  \item Cross-encoder reranking evaluation for precision-critical use cases
  \item Embedding model fine-tuning or replacement based on drift analysis findings
  \item Integration testing suite with containerized database and mocked AWS services
  \item User feedback loop for continuous retrieval quality improvement
  \item Token budget management: pre-prompt token count check to prevent context truncation in constrained-context-window backends
  \item Secrets Manager / SSM Parameter Store migration for production-grade API key handling
\end{itemize}
